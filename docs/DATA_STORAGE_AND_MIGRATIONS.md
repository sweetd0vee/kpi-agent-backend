# Хранение данных и миграции схемы

Этот документ дополняет [DATABASE.md](./DATABASE.md): как в проекте устроено постоянное хранение и **нужно ли выполнять миграции** при изменении кода.

---

## Как правильно хранить данные в этом проекте

### Где лежат данные

- **Основное хранилище структурированных данных** — **PostgreSQL** (или SQLite для локальных экспериментов), строка подключения задаётся в `DATABASE_URL`.
- **Схема таблиц** описана в SQLAlchemy-моделях (`src/db/models.py`) и создаётся/дополняется через `Base.metadata.create_all()` при вызове `init_db()` (старт приложения или `POST /api/db/init`).
- **Файлы документов** (если используется MinIO) — отдельно, см. [FILE_STORAGE_MINIO.md](./FILE_STORAGE_MINIO.md).

### Соглашения по содержимому

1. **Идентификаторы**
   - Строки целей и справочников (`board_goals`, `leader_goals`, `strategy_goals`, `process_registry`, `staff`): поле `id` — **UUID** (в API и на фронте — строка UUID v4).

2. **Текстовые поля**
   - В моделях — без жёсткой длины; «пусто» в бизнес-логике часто кодируется как **пустая строка `""`**, а не `NULL`, чтобы упростить сериализацию и фронтенд.

3. **ФИО руководителей**
   - Отдельной таблицы `leaders` нет. В `board_goals` и `leader_goals` ФИО хранится в поле `last_name`. Для подсказок в других экранах используется `GET /api/reference/responsibles` (агрегация из `board_goals`, `leader_goals` и `staff.head`), а не отдельная сущность в БД.

4. **Массовые таблицы целей через API**
   - Эндпоинты `PUT /api/board-goals`, `PUT /api/leader-goals`, `PUT /api/strategy-goals` **полностью заменяют** набор строк (типичный паттерн DELETE + INSERT по сути). Это не «миграция», а **замена данных приложением**; бэкап перед массовой правкой — на стороне оператора/процесса.

5. **Целостность**
   - При изменении типов колонок или FK вручную в SQL нужно соблюдать порядок: снять ограничения → изменить типы → вернуть ограничения (как в примере скрипта в репозитории).

## Нужно ли делать миграции?

### Что делает `create_all` автоматически

- Создаёт **отсутствующие** таблицы и объекты, которые входят в `Base.metadata`.
- **Не удаляет** старые таблицы, если модель убрана из кода.
- **Не изменяет** уже существующие таблицы: не добавляет колонки, не меняет типы, не переименовывает поля.

Отдельно от `create_all`, функция **`init_db()`** в `src/db/database.py` после создания таблиц может выполнять **узкие одноразовые правки** под уже существующую БД (например переименование колонки `staff`, удаление таблицы `leaders` и колонки `board_goals.leader_id`). Это не отменяет правила выше для произвольных изменений схемы: при новых колонках в модели по-прежнему нужен явный шаг миграции.

То есть **первый деплой** на пустую БД обычно достаточно описать моделями + `create_all`. **Обновление схемы на БД, где данные уже есть**, это уже отдельная задача.

### Когда без миграции не обойтись

Миграция (изменение схемы с сохранением/переносом данных) **нужна**, если вы:

- добавили/удалили/переименовали колонку в модели;
- изменили тип колонки (например `VARCHAR` → `UUID`);
- добавили/изменили внешний ключ;
- удалили таблицу, которая ещё есть в продакшен-БД.

Иначе код и реальная БД разойдутся: при старте ошибок может не быть, но запросы начнут падать или данные будут записываться неверно.

### Как в этом репозитории принято делать миграции

В проекте **нет встроенного Alembic/Flyway** как обязательной части CI. Исторически использовались:

- **ручной SQL** в консоли `psql` или клиенте;
- **одноразовые Python-скрипты** в `scripts/` (например: переименование `staff.functional_block` → `business_unit` — `scripts/migrate_staff_functional_block_to_business_unit.py`; в `process_registry`: `process_name` → `process`, `owner_full_name_ref` → `leader` — `scripts/migrate_process_registry_process_and_leader.py`; перевод первичного ключа `id` с `VARCHAR` на `UUID` — `scripts/migrate_row_ids_to_uuid.py`). Удаление устаревшей таблицы `leaders` и колонки `board_goals.leader_id` при старте приложения выполняется в `src/db/database.py`; вручную — `scripts/migrate_drop_leaders.py`. Исторический скрипт `scripts/migrate_leaders_uuid.py` относится к старой схеме с `leaders` и при необходимости оставлен только как справка по технике «снять FK → ALTER → вернуть FK».

Рекомендуемый порядок работ при изменении схемы:

1. **Сделать резервную копию БД** (`pg_dump` или снимок тома).
2. Проверить скрипт на **копии** базы.
3. Зафиксировать в репозитории **точный SQL или скрипт** миграции и краткое описание в коммите / в этом разделе docs.
4. После миграции перезапустить приложение и проверить `GET` эндпоинты и критичные сценарии.

### Когда имеет смысл внедрить Alembic (или аналог)

Имеет смысл, если:

- несколько сред (dev/stage/prod) и нужна **повторяемая** история изменений;
- часто меняется схема и важна **ревизия** миграций в Git.

До внедрения инструмента миграций достаточно **дисциплины**: модель в `models.py` = источник истины для *новых* установок; для *существующих* БД — явный шаг миграции + бэкап.

## Краткий чеклист

| Ситуация | Действие |
|----------|----------|
| Новая пустая БД | Запуск бэкенда или `POST /api/db/init` — `create_all` создаст таблицы. |
| Новая таблица в коде, БД пустая или таблицы ещё нет | Достаточно деплоя с новым кодом (create_all создаст таблицу). |
| Изменили колонку/тип/FK в существующей БД | Нужна **миграция** (SQL или скрипт), не полагайтесь только на `create_all`. |
| Удалили модель таблицы | `create_all` **не удалит** таблицу; удалите вручную (`DROP TABLE ...`), если она больше не нужна. Для `leaders` / `leader_id` в `board_goals` миграция встроена в `init_db` (см. `database.py`). |
